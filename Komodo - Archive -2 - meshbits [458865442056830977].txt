[24-Jul-20 03:15 PM] grewalsatinder#9653 (pinned)
the steps are:
- Download the build from release
- Extract the downloaded archive
- make config.json as explained in ReadMe
- open command line terminal or `cmd` on windows or `Power Shell` on windows
- Navigate to the directory where you extracted the Shurli files
- start it by executing `./shurli start`, which will perform some actions such as:
  - start Shurli as a backend service.
  - start's `SHURLI0` DEXP2P smart chain specifically created for this Shurli application.
  - opens the browser address of Shurli's web interface automatically so that user doesn't has to do it manually. Port will be randomly selected for Shurli web app after identifying any available free port on machine.

To stop:
- Just navigate back to the directory where you extracted the shurli files and execute the command `./shurli stop`. It will stop both `SHURLI0` smart chain, and kill the backend `shurli` service.

[19-Sep-20 05:21 AM] Decker#3767
@grewalsatinder use following way to cross-compile for Windows:
```
git clone https://github.com/satindergrewal/saplinglib
cd saplinglib
curl https://sh.rustup.rs -sSf | sh
source $HOME/.cargo/env
rustup target add x86_64-pc-windows-gnu
cargo build --target=x86_64-pc-windows-gnu --release
x86_64-w64-mingw32-gcc ./src/C/main.c -I./src -L./target/x86_64-pc-windows-gnu/release -lsaplinglib -lws2_32 -luserenv -o main.exe
```

the first PoC made it work via Golang + HTML interface
the second PoC is a little bit improvement on user experience part, as it also generates a shielded address based on seed phrase
also without requiring to have any blockchain daemon running, just using the code functions.
the better target is to move this user interface to Flutter based UI, for which I'm working with a designer who is in the process of making wireframe of the UI/UX
once that is finalised I'll move my efforts on making the UI in Flutter based on these designs
The basic target is to make a the these subatomic swaps automated for big amounts, which gets traded in small chunks and managed in case the trade fails for N reasons.
the later target is to get jl777's new subatomic swaps based JUMBLR coded in Shurli
and even farther target is to port the whole functionality of the C code based subatomic to Golang based subatomic, so that I can extend it further to add more or many types of cryptocurrencies

the software architecture design is a little difference in Shurli
For Flutter UI based Shurli development I divided it in 2 parts
Shurli-Server
Shurli-Client(s)


[27-Sep-20 07:31 PM] grewalsatinder#9653
Shurli-Server is golang based gRPC code, which interacts with subatomic API, and any/all cryptocurrencies on the machine to make trades etc. I'll probably extend it to manage these wallets as well so it will be multi-currency wallet manager too in future.
Shurli-Client(s) coded in Flutter is planned to be a single code, which compiles for all supported platforms. Including Windows/Mac/Linux/iOS/Android, and even Web Browser!!
So, until then my designer is busy making the UI/UX designs, I'm trying to work things out in Shurli-Server part
making things improve and ready for the user action flow process via gRPC, so that the clients which connect to the shurli-server can do all required actions from the GUI
my idea is to make it possible to host Shurli-Server anywhere user wants.
Even on their home laptops/desktop or small servers at home.
Or if they want to host it on cloud VPS/Server etc.
and allow them to connect to Shurli-Server via Shurli-Client application from anywhere, any device possible, including mobiles.
Having this architecture will allow the users/trades to turn off their client applications after putting in their trades
and the trading server, the Shurli-Server can be left running 24/7
so all things automation of trading etc can be planned for it.
The expected troubles would be to manage the multi-currencies in automated ways. But I think it has to be taken as a challenge to manage.
Some times there needs to be human interaction needed to sort things out with the full blockchain daemons etc. But I think many of these can be put to mange via graphical interface.
where, things are not reliant 100% on automation, but a level of semi-automation.
This is a lot and lots of work!
And I'm using the tools many of which are still new to me, much like Flutter/Dart programming language, and to some extent Golang too, but I'm getting better with Golang as I'm coding things day to day.

[20-Dec-20 06:09 PM] grewalsatinder#9653
I should explain a bit on my idea of Shurli's core. It is this:
**1. Shurli's Core:** A framework, which allows using the go language libraries like `kmdgo` (KMD RPC API and utilities for managing handling smart chain actions), `saplinglib` (creating and managing of zcash sapling address) and any new go language libraries.


[20-Dec-20 06:11 PM] grewalsatinder#9653
**2. Shurli-Server:** The server which is built using Shurli's core framework, using Go language. It is a gRPC API service provider. Shurli server links directly with `komodod`, or other blockchain daemons directly. Also, connects with the `subatomic` binary/daemon. Receives request from _Shurli-Client(s)_ to perform actions like wallets actions (`sendtoaddress`, `listtransactions` etc.). Blockchain daemons like `komodod` and any other DEX specific binaries like `subatomic` are located along or on the same server where _Shurli-Server_ is installed, so that _Shurli-Server_ executes these wallet and DEX actions.


[20-Dec-20 06:14 PM] grewalsatinder#9653
**3. Shurli-Client(s):** The client that I worked on side by side already is a Flutter code based client, compilable to both Desktop (MacOS, Linux, Windows) and Mobile (iOS and Android), connects to a gRPC of _Shurli-Server_. _Shurli-Client_'s purpose would to be the graphical interface for end users, which sends request to the server, and let the server do it's operations. _Shurli-Client(s)_ can be closed/turned off/shutdown after sending some actions to _Shurli-Server_. For example sending a transaction request to server, or if a `subatomic` DEX module is built and installed on _Shurli-Server_, then setup a trade ask/bid via _gRPC API_ on _Shurli-Server_, and close the _Shurli-Client_ application. It means the end user doesn't has to always open the client app after placing trade.

Benefit of Shurli-Server_ is it can be installed on a Home server, a cloud server or VPS etc. The _Shurli-Server_ has to be installable and manageable by each user themselves, so it'll be made with semi-automation design thoughts, coz past experience tells blockchain managed on nodes can not be fully automated, or at least it needs extensive gradual enhancement in automation.

Once this **Shurli framework**, server/client is coded, I can also chose to start coding a `subatomic` DEX module and a **Jumblr module** for Shurli based framework. And can proceed with legal stuff if I need to at that time, if really required.
